# K-Style 합성 파이프라인 분석 메모

## 결론

이 분석은 이 프로젝트에 꽤 직접적으로 맞는다.

현재 저장소에도 실제로 아래 다단계 API 구조가 보인다.

- `app/api/hair`
- `app/api/outfit`
- `app/api/bgremove`
- `app/api/iclight`

즉, "헤어 변경 -> 의상 변경 -> 배경 제거 -> 조명 보정/합성" 같은 멀티스텝 생성 파이프라인 리스크를 이미 안고 있는 구조로 볼 수 있다.

`md-doc/ui-flow.md` 기준으로 보면 실제 사용자 흐름도 아래처럼 다단계다.

- 2단계: 헤어 합성 + 백그라운드 전신 생성
- 3단계: 의상 합성
- 4단계: 배경 제거 + 배경 합성
- 5단계: 선택형 IC-Light 보정

즉 단순한 "기능이 많다" 수준이 아니라,
사용자 대기 흐름 자체가 여러 AI inference와 폴링에 걸쳐 연결된 구조다.

---

## 1. 가장 치명적인 문제

### 1) 누적 열화(Uncanny Valley 누적)

- 이 프로젝트에 가장 치명적일 가능성이 크다.
- 각 단계가 독립적으로 돌면, 앞 단계에서 깨진 얼굴/톤/질감이 뒤 단계에서 더 심하게 무너질 수 있다.
- 무료 실험이면 버틸 수 있어도, 결제형 서비스라면 환불과 불만으로 바로 이어질 수 있다.

한 줄 판단:

- `hair -> outfit -> bgremove -> iclight` 식으로 갈수록 결과물 누적 열화 위험이 높다.

### 2) 대기 시간과 병목

- 이것도 매우 현실적인 문제다.
- API를 여러 번 핑퐁하면, 중간 한 단계만 느려도 전체 UX가 무너진다.
- 모바일 환경에서는 특히 치명적이다.

`ui-flow.md`의 추정치를 그대로 보면:

- 헤어: 30~60초
- 의상: 30~60초
- 합성: 20~40초
- IC-Light: 30~60초
- 전체 합계: 약 90~180초

이건 "AI라서 조금 기다리는 수준"이 아니라,
결제형 모바일 웹앱 기준으로는 상당히 위험한 대기 시간이다.

한 줄 판단:

- 단계 API 분리 구조는 품질뿐 아니라 속도와 실패율에서도 불리하다.

---

## 2. 조건부로 매우 중요한 문제

### 3) 인종/피부톤 붕괴

- 해외 유저가 실제 타겟이라면 반드시 대비해야 한다.
- K-style 생성은 얼굴 골격이나 피부톤을 모델이 임의로 바꿔버릴 위험이 있다.
- 이건 단순 취향 문제가 아니라 해외 반발 이슈로 번질 수 있다.

한 줄 판단:

- 해외 결제 유저를 받을 거면, 피부톤/얼굴 정체성 유지 장치는 필수다.

### 4) 배경 합성 티와 빛 부조화

- `iclight`가 있어도 완전히 해결되지는 않을 가능성이 높다.
- 사용자가 형광등 아래 셀카를 올리면, 한낮 궁궐 배경과의 부조화가 쉽게 난다.
- 업로드 가이드와 전처리 규칙이 같이 있어야 한다.

한 줄 판단:

- 모델 하나로 해결하려 하지 말고, 입력 사진 가이드도 제품 설계에 포함해야 한다.

---

## 3. 전략 방향에 대한 의견

가장 동의하는 부분:

- "단계를 잘게 쪼개는 합성 앱"보다
- "얼굴 정체성만 유지하고 한 번에 새로 렌더링하는 원샷 생성" 쪽이
- 품질 면에서는 훨씬 유리하다.

즉:

- 현재 구조는 구현은 쉬워 보여도
- 장기적으로는 품질, 속도, 클레임 면에서 더 위험할 수 있다.

한 줄 결론:

- 이 프로젝트에서는 `멀티스텝 합성`보다 `정체성 유지형 원샷 생성` 방향이 더 안전하다.

2. `다단계 병목`

3. 피부톤/인종 정체성 유지
4. 조명/합성 자연스러움

이 프로젝트는 지금 구조대로 가면 "돌아는 가지만 결과물 퀄리티와 대기 시간 때문에 결제형 서비스로는 위험한 멀티스텝 파이프라인"일 가능성이 높다.

## MVP 운영안

### 1. 지금 당장 버려야 할 기대

- "헤어, 의상, 배경, 조명까지 모두 완벽하게 통제되는 합성 서비스"를 MVP 단계에서 약속하면 안 된다.
- 기능이 많아 보일수록 실패 지점도 늘어난다.
- MVP는 "유저 얼굴 정체성을 최대한 유지하면서, K-style 결과물 1장을 빠르게 안정적으로 받게 하는 것"에만 집중해야 한다.

한 줄 원칙:

- 옵션이 많은 서비스보다, 결과가 안정적인 서비스가 MVP에 더 맞다.

### 2. MVP의 가장 안전한 제품 정의

권장 MVP:

- 유저 사진 1장 업로드
- 스타일 프리셋 선택
  - 예: `한복 궁궐 무드`
  - 예: `홍대 네온 스트릿`
  - 예: `가로수길 카페 무드`
- 서버에서 가능한 한 적은 수의 inference로 생성
- 결과 이미지 1장 반환

이때 중요한 점:

- 유저가 헤어, 의상, 배경을 각각 따로 고르게 하지 않는다.
- 프리셋 내부에서 한 번에 묶여 생성되도록 한다.
- 즉 `UploadFlow -> HairFlow -> OutfitFlow -> LocationFlow -> DoneFlow` 전체를 다 노출하는 서비스보다,
  `업로드 -> 스타일 선택 -> 결과`처럼 보이는 서비스가 훨씬 안전하다.
- "합성 툴"보다 "스타일 결과물 생성기"처럼 보여 주는 편이 더 안전하다.

### 3. MVP에서 꼭 필요한 안전장치

#### A. 업로드 가이드 강화

- 정면 사진
- 얼굴 가림 최소
- 강한 그림자 없음
- 과한 필터 없음
- 해상도 하한선 설정

이건 선택이 아니라 필수다.
입력이 나쁘면 뒤 단계 모델이 아무리 좋아도 결과가 무너진다.

#### B. 실패 허용 UX

- 결과 생성 중 대기 시간을 명확히 보여 준다
- 실패 시 다시 시도 가능하게 한다
- "완벽한 원본 보존"을 약속하지 않는다
- 결과가 다를 수 있다는 안내를 미리 준다

#### C. 스타일 프리셋 제한

- 처음에는 3~5개 프리셋만 운영
- 프리셋마다 배경, 의상, 무드를 고정
- 성공률 낮은 프리셋은 바로 비활성화

한 줄 원칙:

- 자유도보다 성공률을 우선한다.

## 최소로 할 수 있는 방법

### 방법 1. 현재 구조를 최대한 유지하는 최소안

가장 현실적인 최소안은 아래다.

- `hair`, `outfit`, `bgremove`, `iclight` 전부를 유저 선택형으로 노출하지 않는다
- 내부적으로는 쓰더라도 프론트에는 `스타일 프리셋 선택`만 보여 준다
- 실패율 높은 단계는 과감히 끈다
- `DoneFlow`의 선택형 `IC-Light`는 MVP에서는 기본 OFF 또는 제거가 안전하다
- `body/extend`처럼 백그라운드 생성도 성공률이 낮으면 먼저 끌 수 있어야 한다
- 우선은 `배경 제거 + 통합 생성` 또는 `원샷 생성` 중심으로 단순화한다

한 줄로:

- "많이 선택하게 하는 앱"이 아니라 "적게 선택해도 결과가 잘 나오는 앱"으로 줄인다.

### 방법 2. 더 나은 최소안

가능하면 아래 구조가 더 낫다.

- 유저 얼굴 특징만 유지
- 프리셋 프롬프트로 헤어/의상/배경을 한 번에 생성
- 최종 이미지만 반환

즉:

- 합성(Composite) 단계 여러 개를 노출하지 않고
- inference 1회로 끝내는 구조

이 방식의 장점:

- 품질 누적 열화가 줄어든다
- API 병목이 줄어든다
- 실패 원인 추적이 쉬워진다

### 방법 3. 정말 최소 MVP

정말 최소로 간다면:

- 헤어 변경 제거
- 의상 개별 변경 제거
- 배경 선택도 몇 개 프리셋으로만 제한
- 배경 제거/IC-Light를 후처리 옵션으로 빼거나 MVP에서는 제거
- `얼굴 유지 + 프리셋 스타일 1장 생성`만 제공

즉 MVP는:

- 업로드
- 프리셋 선택
- 결과 1장

이 세 단계만 남기는 것이 가장 안전하다.

## 실무적 추천안

현재 프로젝트 기준 추천 순서는 아래다.

1. 유저 선택 옵션 축소
2. 업로드 품질 가이드 강화
3. 프리셋 수 제한
4. `Hair -> Outfit -> Location -> Done` 전체 폴링 체인을 최대한 줄이기
5. 장기적으로는 원샷 생성 구조로 전환

추가 실무 메모:

- `ui-flow.md` 기준 현재 구조는 "백그라운드에서 동시에 돌아가는 작업"이 많다.
- 이 구조는 체감상 빨라 보이게 만들 수는 있지만,
  실제로는 실패 원인 추적과 복구가 더 어려워진다.
- MVP에서는 병렬 최적화보다, 성공률 높은 단순 흐름이 우선이다.

---

## 최종 한 줄

이 프로젝트의 MVP는 "강력한 합성 툴"이 아니라, "실패율이 낮은 K-style 결과물 생성기"로 정의해야 가장 안전하다.

## 가능한 툴 / API 추천

`ui-flow.md`와 현재 저장소 구조를 기준으로 보면, 지금 필요한 건 "기능이 많은 API"보다 "단계를 줄일 수 있는 API"다.

### 1. 가장 추천하는 방향: `fal` 중심 단순화

이유:

- 공식 문서상 `fal`은 FLUX 계열 image editing 모델을 빠르게 제공한다.
- `FLUX.1 Kontext`는 이미지 편집, 로컬 수정, character consistency 쪽을 강하게 밀고 있다.
- `FLUX.2`는 더 높은 품질과 다중 참조, 색상 제어(HEX) 같은 기능을 제공한다.
- `Queue + Webhook` 구조가 공식 지원돼서, 지금처럼 프론트에서 3초 폴링을 길게 돌리는 구조를 줄이기 좋다.

적용 포인트:

- `hair`, `outfit`, `bgremove`, `composite`를 따로 유저에게 노출하지 말고
- 서버에서 `fal`의 image editing 계열 모델 1회 또는 최소 횟수로 처리
- 프론트는 `업로드 -> 프리셋 선택 -> 대기 -> 결과`만 보이게 바꾼다

실무 한 줄:

- 가장 현실적인 "줄이기"는 `fal` 쪽 image editing 모델 하나로 최대한 통합하는 방향이다.

### 2. 차선책: `ComfyUI Cloud` 또는 자체 ComfyUI 워크플로우

이유:

- 공식 문서상 ComfyUI는 Cloud API와 OpenAPI 스펙을 제공한다.
- 워크플로우 전체를 하나의 API처럼 실행할 수 있다.
- 지금처럼 `hair -> outfit -> bgremove -> composite`를 개별 엔드포인트로 쪼개지 않고, 내부 노드 그래프 하나로 묶을 수 있다.

적용 포인트:

- 현재 단계별 API를 유지하되, 백엔드에서는 하나의 ComfyUI workflow job만 실행
- 결과는 WebSocket 또는 webhook으로 받는다
- 프론트는 단일 job 상태만 받는다

실무 한 줄:

- "파이프라인은 유지하되 API는 하나로 줄이고 싶다"면 ComfyUI가 가장 잘 맞는다.

### 3. 현재 구조 유지 시 현실적 개선: `Replicate/FAL webhook` 전환

이유:

- Replicate 공식 문서에 webhook 지원이 있다.
- FAL 공식 문서에도 queue + webhook 지원이 있다.
- 지금 프로젝트는 `3초 간격 * 최대 40회` 폴링이 여러 단계에 걸쳐 반복되는데, 이걸 webhook 기반으로 바꾸면 프론트 병목이 줄어든다.

적용 포인트:

- 프론트가 각 단계별 `/poll`을 계속 때리지 않게 한다
- 백엔드가 webhook을 받아 store/db에 최종 상태를 저장
- 프론트는 한 개의 `/job/status`만 조회

주의:

- Replicate는 공식 문서상 API prediction의 input/output 데이터가 기본적으로 오래 보존되지 않을 수 있으니, webhook 수신 후 결과를 바로 저장하는 구조가 필요하다.

실무 한 줄:

- 지금 구조를 당장 못 버리면, 최소한 폴링 체인부터 webhook 중심으로 바꿔야 한다.

## 이 프로젝트에 맞는 실제 적용 순서

### 추천안 A. 가장 안전한 MVP

1. 유저 업로드
2. 스타일 프리셋 선택
3. 서버에서 단일 image edit/generation job 실행
4. 결과 1장 반환

이때 추천:

- 1순위: `fal`의 FLUX 계열 image editing
- 보조 선택지: ComfyUI workflow 1회 실행

버릴 것:

- 헤어 개별 선택 UI
- 의상 개별 선택 UI
- 배경 선택 후 별도 합성 UI
- DoneFlow의 IC-Light 선택 보정

즉:

- 지금 `5단계 UI`를 `2~3단계 UX`로 줄이는 것이 핵심이다.

### 추천안 B. 지금 코드베이스를 최대한 살리는 최소 수정안

1. `UploadFlow` 유지
2. `HairFlow`, `OutfitFlow`, `LocationFlow`를 사용자 선택 단계가 아니라 내부 프리셋 로직으로 축소
3. 실제 외부 호출은 서버 내부에서만 수행
4. 프론트는 단일 "생성 중" 상태만 본다

실제 구현 방향:

- `/api/create/run` 같은 단일 엔드포인트를 만든다
- 내부에서 `fal` 또는 `ComfyUI`로 1회 job 제출
- 완료는 webhook으로 받는다
- 프론트는 `jobId` 하나만 저장한다

이 방식의 장점:

- 현재 Zustand store를 완전히 버리지 않고도 축소 가능
- UI는 유지하되 로직만 단순화 가능

## 최소로 할 수 있는 기술 조합

### 조합 1. 가장 추천

- 생성/편집: `fal` FLUX Kontext 또는 FLUX.2 image editing
- 비동기 처리: `fal queue + webhook`
- 프론트 상태: 단일 job status

장점:

- 빠름
- webhook 지원 좋음
- "한 번에 편집" 방향에 잘 맞음

### 조합 2. 제어력이 더 필요한 경우

- 실행 엔진: `ComfyUI Cloud API` 또는 자체 ComfyUI
- 워크플로우: face 유지 + 스타일 프리셋 + 배경/조명 한 번에 처리
- 완료 수신: websocket/webhook

장점:

- 파이프라인 제어력 높음
- 나중에 모델 바꾸기 쉬움

단점:

- 초기 설계 난이도 높음

### 조합 3. 당장 급한 응급처치

- 현재 Replicate/FAL 단계 API 유지
- polling 제거 또는 축소
- webhook + 서버 오케스트레이션으로 변경
- IC-Light는 MVP에서 제거

장점:

- 기존 코드 재사용 가능

단점:

- 근본적으로는 멀티스텝 누적 열화 문제가 남음

## 최종 추천

이 프로젝트에서 과정을 줄이고 싶다면 우선순위는 아래가 맞다.

1. `프론트 5단계 UX`를 줄인다
2. `단일 job API`로 바꾼다
3. `webhook` 기반으로 바꾼다
4. 가능하면 `fal` image editing 또는 `ComfyUI workflow 1회 실행`으로 통합한다

한 줄 결론:

- 지금 프로젝트에 가장 현실적인 해법은 `fal 중심 단일 생성 job` 또는 `ComfyUI 단일 workflow job`으로 줄이고, 프론트는 `업로드 -> 프리셋 선택 -> 결과`만 남기는 것이다.
