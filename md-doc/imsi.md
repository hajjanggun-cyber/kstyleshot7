# 추가 메모: 헤어 합성 및 최종 합성 대기 멘트 방향

로딩 문구는 단순한 상태 표시가 아니라, 유저가 기다리는 시간을 가치 있게 느끼게 만드는 UX 요소로 봐야 한다.

특히 아래 두 구간에서 중요하다.

- 헤어 합성 대기 구간
- 최종 의상/배경 합성 대기 구간

## 기본 원칙

- 기술 상태를 기계적으로 설명하지 않는다.
- 결과가 더 좋아지고 있다는 느낌을 줘야 한다.
- 무엇은 유지되고 무엇이 바뀌는지를 짧게 알려줘야 한다.
- 과장된 AI 톤보다 차분하고 고급스러운 브랜드 톤이 낫다.

## 헤어 합성 멘트 방향

유저가 느껴야 하는 메시지:

- 얼굴은 유지하고 있다
- 헤어만 자연스럽게 맞추고 있다
- 곧 결과가 완성된다

예시 방향:

- 메인: `얼굴선에 맞춰 헤어를 정교하게 맞추는 중`
- 서브: `원래 인상은 그대로 두고, 선택한 스타일만 자연스럽게 입히고 있어요`

## 최종 의상/배경 합성 멘트 방향

유저가 느껴야 하는 메시지:

- 선택한 헤어는 그대로 유지하고 있다
- 의상, 배경, 분위기를 한 장으로 정리하고 있다
- 단순 생성이 아니라 화보를 완성하고 있다

예시 방향:

- 메인: `선택한 무드로 화보를 완성하는 중`
- 서브: `헤어는 유지한 채 의상, 배경, 분위기를 한 장의 결과로 정리하고 있어요`

## 제품 관점 메모

- 로딩 문구는 `AI 처리 중`보다 `큐레이션 중`, `완성 중`, `정리 중` 같은 표현이 더 좋다.
- 헤어 단계와 최종 단계의 대기 멘트는 서로 달라야 한다.
- 둘 다 짧고 직관적이어야 하며, 읽자마자 무슨 작업이 진행 중인지 이해돼야 한다.

# 추가 메모: 경복궁 한복 테스트 프롬프트 및 현재 연결 API

## 테스트 프롬프트 보정 방향

- 얼굴은 반드시 유지한다.
- 헤어스타일, 헤어 컬러, 앞머리, 전체 헤어 실루엣은 반드시 유지한다.
- 셀카 구도는 `가슴 위까지 보이는 근접 셀카`로 고정한다.
- 손에 든 스마트폰은 실제 셀카처럼 보이게 하고, 케이스와 후면 카메라 렌즈가 보이도록 지시한다.
- 의상은 `젊은 아이돌 무드의 세련된 한복 저고리`로 구체화한다.
- 배경은 `경복궁 마당 + 골든아워 + 뒤쪽 관광객이 소프트 블러 처리된 상태`로 구체화한다.
- 전체 결과는 `포토리얼한 K-pop 아이돌 셀카 스냅` 느낌으로 유도한다.

## 현재 테스트용 참조 이미지

- `/public/reference-templates/1-hanbok-1.jpeg`
- `/public/reference-templates/1-hanbok-2.jpeg`

현재는 위 두 장만 테스트용으로 연결되어 있다.

## 현재 연결된 API

### 1. 헤어 생성 API

- `POST /api/hair/preview`
- `GET /api/hair/poll`

설명:

- 헤어 합성 단계에서 사용한다.
- 내부적으로 Replicate의 `flux-kontext-apps/change-haircut`를 호출한다.

### 2. 최종 합성 API

- `POST /api/final/render`
- `GET /api/final/poll`

설명:

- 선택된 헤어 합성 이미지를 베이스 이미지로 사용한다.
- 선택된 참조 이미지에 매핑된 프롬프트를 실행한다.
- 내부적으로 Replicate의 `black-forest-labs/flux-kontext-pro`를 호출한다.

## 현재 실제 연결 구조

1. 유저가 헤어 2개를 선택한다.
2. 시스템이 헤어 결과 2장을 생성한다.
3. 유저가 그중 1장을 선택한다.
4. 유저가 경복궁 한복 테스트 참조 이미지 2장 중 1장을 선택한다.
5. 시스템이 해당 참조 이미지에 연결된 프롬프트를 `flux-kontext-pro`로 전송한다.
6. 최종 결과를 폴링해서 완료 화면에 보여준다.

# 추가 메모: 참조 이미지 선택형 최종 합성 구조

- 유저는 최종 스타일을 직접 텍스트로 입력하지 않고, 미리 준비된 참조 이미지 카드 중 하나를 선택한다.
- 각 참조 이미지 카드에는 서버에서 관리하는 고정 프롬프트가 매핑되어 있다.
- 최종 렌더링의 베이스 이미지는 항상 `헤어 합성 완료 이미지`로 고정한다.
- 최종 모델은 선택된 참조 이미지에 대응하는 의상, 배경, 포즈 프롬프트만 적용한다.

## 핵심 구조

1. 유저가 헤어 스타일 2개를 선택한다.
2. 시스템이 헤어 합성 결과 2장을 생성해 유저에게 보여준다.
3. 유저는 생성된 2장 중 최종 1장만 선택한다.
4. 선택된 1장만 최종 렌더링의 베이스 이미지가 된다.
5. 이후 유저가 참조 이미지 썸네일을 고른다.
6. 시스템은 해당 카드에 연결된 프롬프트 템플릿을 불러온다.
7. 최종 모델이 의상, 배경, 포즈를 반영해 화보 1장을 생성한다.

## 운영 원칙

- 헤어는 총 2개까지 선택할 수 있다.
- 시스템은 선택된 2개의 헤어에 대해 결과 2장을 생성한다.
- 유저는 생성된 2장 중 오직 1장만 선택할 수 있다.
- 선택된 1장만 최종 합성 단계로 넘어간다.
- 유저는 참조 이미지를 고른다.
- 시스템은 그 이미지에 연결된 프롬프트를 실행한다.
- 베이스는 항상 헤어 합성 완료 이미지로 유지한다.

## 제품 원칙

- 헤어 합성 단계는 `2개 생성 -> 1개 선택 -> 최종 합성 이동` 구조로 고정한다.
- 참조 이미지는 선택 UI 용도다.
- 실제 모델 입력은 `참조 이미지 자체`보다 `참조 이미지에 매핑된 프롬프트` 중심으로 운영한다.
- 최종 프롬프트에는 반드시 얼굴과 헤어를 유지하라는 지시가 들어가야 한다.

# 헤어 합성 결과 유지 + 의상/배경/포즈 텍스트 합성 구조

대표님께서 제안하시고 클로드가 동의한 `헤어 합성 사진 베이스 + 의상/배경/포즈 텍스트 합성` 구조가 현시점에서 가장 완성도 높은 밸런스입니다.

이 문서는 왜 이 구조가 최적해인지, 그리고 실제 서비스 플로우를 어떻게 설계해야 하는지를 기획자의 언어로 정리한 메모입니다.

## 핵심 결론

- 헤어는 먼저 별도 렌더링으로 확정한다.
- 유저는 헤어 옵션 2개를 선택하고, 시스템은 결과 2장을 생성한다.
- 유저는 생성된 헤어 결과 2장 중 최종 1장만 선택한다.
- 선택된 헤어 합성 결과 1장을 베이스 이미지로 고정한다.
- 최종 렌더링에서는 헤어를 다시 건드리지 않는다.
- 최종 렌더링은 의상, 배경, 포즈만 텍스트로 통제해서 1장 완성본을 생성한다.

즉, 구조는 아래 한 줄로 요약된다.

`헤어 2개 생성 -> 1개 선택 -> 베이스 픽스 -> 의상/배경/포즈 최종 합성`

## 왜 이 구조가 가장 좋은가

### 1. 렌더링 횟수가 딱 2회라서 얼굴 열화가 적다

다단계 합성이 깊어질수록 얼굴 질감이 무너지고, 전체 인상이 어색해지는 언캐니 밸리 문제가 생긴다.

하지만 이 구조는 렌더링을 딱 두 번만 사용한다.

1. 헤어 렌더링
2. 최종 화보 렌더링

즉, 얼굴을 여러 번 다시 굽지 않기 때문에 유저 원본 인상의 보존율이 높다.

### 2. AI의 인지 부하를 줄여준다

최종 단계에서 AI가 동시에 얼굴, 헤어, 의상, 배경, 포즈까지 모두 처리하려고 하면 실패 확률이 급격히 올라간다.

이 구조에서는 헤어 선택 단계에서 이미 `얼굴 + 원하는 머리스타일`이 완전히 고정된다.

따라서 최종 단계의 AI는 헤어를 고민할 필요가 없다.

최종 모델은 아래 세 가지에만 집중하면 된다.

- 의상 표현
- 배경 연출
- 포즈와 화보 분위기 완성

결과적으로 의상 디테일과 전체 장면 완성도가 더 높아진다.

### 3. 사용자는 간단하게 느끼고, 운영자는 강하게 통제할 수 있다

유저 입장에서는:

- 헤어 프리뷰 2장 중 1장 선택
- 참조 이미지 카드 선택

정도로 매우 단순하다.

반면 서버 뒤에서는:

- 대표님이 미리 설계한 의상 프롬프트
- 배경 프롬프트
- 포즈 프롬프트

를 통제된 방식으로 조합할 수 있다.

즉, UX는 단순하지만 결과물은 정교하게 통제할 수 있는 구조다.

## 제안 플로우

### STEP 1. 헤어 렌더링

입력:

- 유저 셀카 1장
- 유저가 고른 헤어 옵션 2개

출력:

- 헤어 프리뷰 2장

방식:

- 병렬 생성
- 생성된 2장을 유저에게 보여줌
- 유저는 비교 후 더 마음에 드는 1장만 선택

이 단계의 목적은 `얼굴 + 헤어`를 가장 먼저 안정적으로 고정하고, 최종 합성에 사용할 단 하나의 베이스 이미지를 고르는 것이다.

### STEP 2. 선택 및 픽스

유저는 헤어 프리뷰 2장 중 최종 1장만 선택한다.

이 시점부터 선택된 이미지는 베이스 이미지가 된다.

이 베이스 이미지에는 이미 다음 요소가 안정적으로 들어 있다.

- 유저 얼굴
- 원하는 머리스타일
- 기본 인상과 분위기

즉, 여기서 `헤어 합성 결과는 유지`와 `선택된 1장만 다음 단계로 이동`이 제품 원칙으로 확정된다.

### STEP 3. 최종 렌더링

입력:

- STEP 2에서 픽스된 헤어 프리뷰 1장
- 통제된 의상 텍스트
- 통제된 배경 텍스트
- 통제된 포즈 텍스트

출력:

- 최종 화보 1장

이 단계에서는 헤어를 다시 생성하지 않는다.

최종 모델은 베이스 이미지 위에서 아래 요소만 확장하거나 완성한다.

- 의상
- 배경
- 포즈
- 화보 톤

## 이 구조가 사실상 최적해인 이유

### 얼굴 보존과 스타일 완성도의 밸런스가 가장 좋다

얼굴 보존만 지나치게 우선하면 결과물이 밋밋해지고, 스타일 완성도만 우선하면 유저 얼굴이 무너진다.

이 구조는 두 목표의 균형점이 가장 좋다.

- 헤어 단계에서 얼굴 보존
- 최종 단계에서 스타일 완성

즉, 보존과 연출을 분리해서 처리하는 구조다.

### 최종 단계 실패 원인이 명확해진다

기존의 복잡한 다단계 구조는 어디서 결과가 깨졌는지 추적하기 어렵다.

하지만 이 구조는 단순하다.

- 헤어가 깨지면 STEP 1 문제
- 의상, 배경, 포즈가 어색하면 STEP 3 문제

즉, 디버깅과 개선 포인트가 분리된다.

### 비용과 속도 측면에서도 유리하다

렌더링이 2회로 제한되면:

- 호출 비용이 과도하게 늘어나지 않고
- 대기 시간이 짧아지고
- 운영 복잡도도 낮아진다

제품 초반에는 특히 이 단순함이 중요하다.

## 성공을 가를 핵심 디테일

이 구조의 성패는 `큐레이션 프롬프트`에 달려 있다.

유저에게 자유 텍스트 입력을 허용하면 다음 문제가 발생한다.

- 품질 편차가 심해진다
- 모델이 이해하기 어려운 문장이 들어온다
- 의상 결과가 무너지기 쉽다
- 실패 케이스가 폭발한다

또한 유저에게 복잡한 의상 이미지를 추가 입력하게 하면 UX가 무거워지고 오류도 늘어난다.

따라서 가장 좋은 방식은 아래와 같다.

- 대표님이 10개 안팎의 완성도 높은 의상 프롬프트 템플릿을 직접 설계한다
- 유저는 텍스트가 아니라 보기 좋은 참조 이미지 카드만 고른다
- 서버는 선택된 카드에 대응하는 영문 프롬프트를 최종 모델로 전송한다

즉, 프론트는 `선택형 UI`, 백엔드는 `통제형 프롬프트 엔진` 구조로 가야 한다.

## 큐레이션 프롬프트 예시

### 예시 1. 유저 화면 라벨

`Y2K 힙스터`

서버 내부 프롬프트 예시:

`A stylish oversized vintage graphic tee with baggy cargo pants, silver chain necklace, highly detailed streetwear`

### 예시 2. 유저 화면 라벨

`하이틴 프레피`

서버 내부 프롬프트 예시:

`A cute pleated plaid mini skirt with a fitted cropped polo shirt and a matching cardigan tied over the shoulders`

핵심은 유저에게는 간단한 카드 선택처럼 보이지만, 실제 뒤에서는 대표님이 깎아둔 정교한 영문 프롬프트가 들어가야 한다는 점이다.

## 제품 설계 원칙

### 유저에게 보여줄 것

- 예쁜 썸네일
- 짧고 직관적인 스타일 이름
- 헤어 프리뷰 2개 비교 선택
- 최종 1개 선택
- 참조 이미지 카드 선택

### 시스템이 내부에서 통제할 것

- 영문 의상 프롬프트
- 영문 배경 프롬프트
- 영문 포즈 프롬프트
- 네거티브 프롬프트
- 결과 일관성을 위한 템플릿 구조

## 실제 구현 플로우

이 구조를 실제 서비스에 연결하면 아래 플로우로 정리할 수 있다.

### 1. 업로드 단계

유저 입력:

- 셀카 1장 업로드

시스템 처리:

- 얼굴 감지
- 기본 리사이즈
- 헤어 합성용 입력 이미지 정규화

스토어 저장값:

- `photoBlobUrl`
- `sessionToken` 또는 작업 세션 ID

### 2. 헤어 선택 단계

유저 입력:

- 헤어 스타일 2개 선택

시스템 처리:

- 헤어 프롬프트 2개를 병렬 호출
- 헤어 프리뷰 2장 생성
- 생성된 2장을 유저에게 노출

스토어 저장값:

- `hair.results[]`
- `hair.predictionIds[]`

### 3. 헤어 픽스 단계

유저 입력:

- 생성된 헤어 프리뷰 2장 중 1장 선택

시스템 처리:

- 선택된 헤어 결과 1장만 베이스 이미지로 확정
- 선택되지 않은 나머지 1장은 최종 합성 입력으로 사용하지 않음

스토어 저장값:

- `hair.picked`
- `hairPreviewUrl`

이 시점부터 이 이미지는 최종 렌더링의 입력 베이스가 된다.

### 4. 스타일 선택 단계

유저 입력:

- 참조 이미지 카드 1개 선택

시스템 처리:

- 카드에 대응하는 내부 프롬프트 템플릿 조회
- 최종 렌더링용 프롬프트 조합

스토어 저장값:

- `selectedReferenceId`

서버 내부 결정값:

- `outfitPrompt`
- `backgroundPrompt`
- `posePrompt`
- `negativePrompt`

### 5. 최종 화보 생성 단계

입력:

- `hairPreviewUrl` 또는 선택된 헤어 결과 이미지
- 선택된 참조 카드에 매핑된 프롬프트

시스템 처리:

- 베이스 이미지는 유지
- 최종 모델은 의상, 배경, 포즈 중심으로 렌더링
- 헤어와 얼굴은 보존 우선으로 처리

출력:

- 최종 화보 1장

스토어 저장값:

- `finalPredictionId`
- `finalImageUrl`

## 화면 기준 UX 플로우

유저가 실제로 느끼는 화면 흐름은 최대한 단순해야 한다.

### 화면 1. 셀카 업로드

- 사진 업로드
- 얼굴 인식 확인
- 다음 단계 이동

### 화면 2. 헤어 선택

- 헤어 옵션 2개 선택
- 생성 버튼 클릭
- 헤어 프리뷰 2장 확인
- 최종 1장만 선택
- 선택된 1장만 다음 단계로 이동

### 화면 3. 참조 이미지 선택

- 참조 이미지 카드 선택
- 완성하기 클릭

### 화면 4. 결과 보기

- 최종 화보 1장 확인
- 다운로드
- 다시 만들기

즉, 유저는 복잡한 설정을 하지 않고도 결과적으로는 정교한 합성 결과를 받게 된다.

## API 관점 제안

현재 구조를 크게 흔들지 않고도 아래 형태로 정리할 수 있다.

### 헤어 생성 API

- `POST /api/hair/preview`
- `GET /api/hair/poll`

역할:

- 헤어 프리뷰 2장 생성
- 결과 폴링

### 헤어 선택 확정 API

- `POST /api/hair/select`

역할:

- 유저가 고른 헤어 프리뷰 1장을 베이스 이미지로 확정

### 최종 렌더링 API

- `POST /api/final/render`
- `GET /api/final/poll`

역할:

- 베이스 헤어 이미지 + 참조 카드 기반 프롬프트로 최종 화보 생성

요청 예시 개념:

- `baseImageUrl`
- `referenceTemplateId`

서버 내부에서는 위 ID를 실제 프롬프트로 변환해서 모델에 전달한다.

## 프롬프트 조합 원칙

최종 프롬프트는 유저 자유 입력이 아니라 템플릿 조합형으로 가야 한다.

구성 예시:

1. 베이스 보존 지시
2. 의상 템플릿
3. 배경 템플릿
4. 포즈 템플릿
5. 화보 톤 템플릿
6. 네거티브 프롬프트

예시 구조:

`Keep the face and hairstyle from the base image. [OUTFIT_PROMPT]. [BACKGROUND_PROMPT]. [POSE_PROMPT]. Editorial k-fashion photoshoot, highly detailed, natural proportions.`

네거티브 프롬프트 예시 방향:

- hair change
- face distortion
- extra fingers
- bad anatomy
- blurry face
- duplicate person
- malformed clothing

## 운영상 중요한 포인트

### 1. 헤어는 무조건 먼저 성공시켜야 한다

이 서비스의 정체성은 결국 `내 얼굴에 원하는 헤어가 자연스럽게 들어간다`는 신뢰감에서 시작된다.

따라서 헤어 단계 품질이 가장 중요하다.

또한 헤어는 2개 생성 후 1개를 고르는 구조여야 한다.

그래야 유저가 비교 선택을 통해 만족도를 높인 뒤, 가장 잘 나온 1장만 최종 렌더링에 태울 수 있다.

### 2. 의상은 템플릿 품질이 승부를 가른다

의상 카드는 많다고 좋은 것이 아니다.

처음에는 정말 잘 나오는 10개 내외로 시작하는 편이 좋다.

### 3. 포즈도 자유 입력보다 큐레이션이 유리하다

포즈를 자유 텍스트로 열어두면 얼굴과 신체 비율이 무너지기 쉽다.

따라서 포즈 역시 카드형 템플릿으로 관리하는 것이 낫다.

### 4. 배경은 실제 장소감이 중요하다

배경은 단순히 장소 이름만 넣기보다, 구도와 시간대까지 포함된 템플릿이 더 안정적이다.

예시:

- `Gyeongbokgung palace courtyard at golden hour`
- `Hongdae street at night with soft neon signs`

## MVP 권장 범위

초기 버전은 아래처럼 좁게 시작하는 것이 좋다.

- 헤어 프리뷰 2장 생성
- 헤어 1장 선택
- 참조 이미지 카드 10개 내외
- 최종 결과 1장 생성

이 정도만으로도 충분히 서비스 가치를 검증할 수 있다.

## 최종 운영 원칙

- 헤어 합성 결과는 반드시 유지한다.
- 헤어는 2장 생성하고, 1장만 선택해서 다음 단계로 넘긴다.
- 최종 렌더는 헤어를 다시 바꾸지 않는다.
- 최종 렌더는 참조 카드에 연결된 의상, 배경, 포즈 프롬프트만 적용한다.
- 자유 입력보다 큐레이션 템플릿을 우선한다.
- 유저 경험은 단순하게, 프롬프트 설계는 백엔드에서 정교하게 관리한다.

## 한 줄 요약

`헤어는 2개 생성해서 1개를 고정하고, 그 1장을 베이스로 참조 카드 프롬프트를 적용해 최종 화보를 만드는 구조가 현재 가장 현실적이고 강한 설계다.`
