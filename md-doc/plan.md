# kstyleshot 실행 계획

## 0. 문서 목적

이 문서는 현재 `kstyleshot` 프로젝트의 제품 방향, 구현 상태, 블로그 운영 원칙, 자산 관리 규칙을 한눈에 정리하는 기준 문서다.

이 문서에서 관리하는 범위는 아래와 같다.

- 제품 구조
- 기술 구조
- 현재 구현 상태
- 우선순위
- 블로그 운영 원칙
- 이미지 및 정적 자산 저장 규칙

## 1. 이미지 및 정적 자산 저장 규칙

이미지 파일은 용도에 따라 저장 위치를 명확히 분리한다. 이 규칙이 지켜지지 않으면 배포 시 경로가 꼬이거나, 임시 파일이 서비스 자산처럼 섞이는 문제가 생긴다.

### 1.1 기본 원칙

- 브라우저가 직접 읽는 파일은 반드시 `public/` 아래에 있어야 한다.
- 작업용 원본 파일은 `image/` 같은 별도 폴더에 둔다.
- 서비스에서 실제로 호출하는 경로와 작업용 보관 경로를 혼용하지 않는다.

### 1.2 폴더별 역할

- `image/`
  - 작업용 원본 파일 보관 위치
  - 디자이너 시안, 전달받은 원본 PNG, 교체 전 참고용 파일 저장
  - 브라우저가 직접 읽는 경로가 아님

- `public/visuals/landing/`
  - 랜딩 페이지에서 직접 노출하는 메인 비주얼 저장
  - 예: `public/visuals/landing/main.png`
  - 브라우저 노출 경로: `/visuals/landing/main.png`

- `public/visuals/blog/`
  - 블로그 인덱스, 카테고리, 포스트 공통용 정적 비주얼 저장

- `public/blog/hero/`
  - 개별 블로그 글의 대표 이미지 저장

- `public/blog/gallery/`
  - 개별 블로그 글의 갤러리 이미지 저장

- `public/backgrounds/`
  - 서비스 내 배경 선택에 직접 쓰는 배경 자산 저장

- `public/samples/`
  - 제품 설명용 샘플 SVG 또는 예시 자산 저장

### 1.3 현재 적용 상태

- 랜딩 메인 이미지는 `image/main.png` 원본을 기준으로 사용한다.
- 실제 서비스 반영 경로는 `public/visuals/landing/main.png` 이다.
- 메인 랜딩 화면과 OG/Twitter 메타 이미지는 같은 파일을 쓰는 구조로 맞춘다.

### 1.4 운영 원칙

- 원본만 `image/`에 넣고 끝내지 않는다. 실제 화면에 반영할 파일은 `public/` 아래로 복사 또는 이동해야 한다.
- 블로그 이미지가 아직 준비되지 않았다면 placeholder를 공개 글에 사용하지 않는다.
- 이미지가 준비되지 않은 글은 텍스트 중심으로 발행하거나 `draft` 상태로 유지한다.

## 2. 제품 정의

`kstyleshot`은 사용자의 사진 1장을 기반으로 K-style 결과를 단계적으로 선택하고 완성하는 서비스다.

기본 흐름은 아래와 같다.

1. 사용자가 사진을 업로드한다.
2. 헤어 결과 2개를 생성한다.
3. 사용자가 2개 중 1개를 고른다.
4. 의상 결과 2개를 생성한다.
5. 사용자가 2개 중 1개를 고른다.
6. 선택된 인물 결과에서 배경 분리를 수행한다.
7. 서울 배경 2개를 합성한다.
8. 사용자가 2개 중 1개를 고른다.
9. 최종 결과를 다운로드한다.

공식 생성 순서는 고정한다.

- 헤어 -> 의상 -> 배경

블로그나 마케팅 문구에서 다른 순서를 설명하더라도 실제 제품 흐름은 바꾸지 않는다.

## 3. 제품과 블로그의 역할 분리

### 3.1 블로그 역할

- 검색 유입 확보
- 장소, 스타일, 촬영, 업로드 준비에 대한 정보 제공
- 허브 글과 세부 글을 통한 내부 링크 구조 형성
- 최종적으로 `create` 흐름으로 연결

### 3.2 제품 역할

- 실제 이미지 생성
- 세션 처리
- 결제 처리
- 생성 상태 추적
- 결과 다운로드 제공

### 3.3 운영 원칙

- 블로그 본문은 정보 제공 중심으로 작성한다.
- 제품 단계 설명은 필요한 범위에서만 다룬다.
- 블로그는 CTA 1개 또는 소수의 자연스러운 연결만 둔다.
- 제품 상세 사양은 블로그보다 별도 제품 문서에서 관리하는 편이 낫다.

## 4. 현재 구현 상태

### 4.1 제품

- Next.js App Router 구조 존재
- 다국어 구조 존재
- `create` 흐름 UI 존재
- 결제, webhook, session status 기본 연결 존재
- 헤어 단계는 기본 흐름이 연결되어 있음
- 의상, cutout, 일부 후속 단계는 추가 점검 필요

### 4.2 블로그

- `/blog` 구조 존재
- EN / KO 분리 구조 존재
- MDX 기반 렌더링 구조 존재
- 현재는 텍스트 중심 발행 구조로 변경됨
- placeholder 이미지는 공개 글에서 제거하는 방향으로 전환함
- 사이트맵 자동 생성 구조 존재

### 4.3 콘텐츠 상태

- 기존 초안 다수는 뼈대 수준이어서 그대로는 SEO 경쟁력이 약함
- 공개 글은 장소 허브형 구조로 재정비가 필요함
- 경복궁 허브 EN / KO는 재작성 시작 상태
- 명동 및 나머지 8개 장소는 허브 우선 작성이 필요함

## 6. SEO 및 콘텐츠 운영 원칙

### 6.1 기본 원칙

- 길기만 한 글은 SEO에 유리하지 않다.
- 검색 의도에 정확히 답해야 한다.
- 제목, 설명, 소제목, 본문이 같은 주제를 일관되게 설명해야 한다.
- 얇은 템플릿 반복형 글은 줄이고, 허브형 설명 글을 먼저 세운다.

### 6.2 언어 운영 원칙

- 영어와 한글은 번역본처럼 쓰지 않는다.
- 각 언어 사용자 검색 의도에 맞춰 별도 문장 구조로 작성한다.
- 한글 글은 설명 밀도와 맥락 연결을 우선한다.
- 영어 글은 검색 결과에서 바로 이해되는 구조를 우선한다.

### 6.3 장소형 콘텐츠 우선순위

서울 10개 장소는 아래 순서로 허브형 전략을 확장한다.

1. 경복궁
2. 명동
3. 남산 N서울타워
4. 가로수길
5. 홍대
6. 한강공원
7. 인사동
8. 이태원/경리단길
9. 잠실 롯데월드
10. SM/HYBE/JYP 사옥 주변

### 6.4 장소형 허브 글에서 반드시 다룰 것

- 왜 유명한가
- 왜 핫플로 소비되는가
- 사람들이 왜 찾는가
- 어떤 상징성을 가지는가
- 다른 장소와 무엇이 다른가
- 사진, 스타일, 관광 콘텐츠와 어떻게 연결되는가

## 7. 현재 우선순위

### 7.1 1순위

- 랜딩 화면 주요 비주얼 정리
- 공개된 허브 글 품질 보강
- 경복궁 허브 EN / KO 정리

### 7.2 2순위

- 명동 허브 EN / KO 작성
- 경복궁 세부 확장 글 설계
- 장소형 내부 링크 구조 정리

### 7.3 3순위

- 나머지 8개 장소 허브 글 순차 작성
- 카테고리/클러스터별 발행 우선순위 재정리
- Google Search Console 제출 및 색인 점검

## 8. 작업 시 주의사항

- 사용자가 지정하지 않은 md 파일을 임의로 만들지 않는다.
- 공개 글에 더미 이미지나 placeholder 자산을 넣지 않는다.
- 인코딩이 깨진 문서는 부분 수정보다 전체 재작성으로 정리하는 편이 안전하다.
- 사용자 요청 없이 코드와 문서 작업을 섞지 않는다.
