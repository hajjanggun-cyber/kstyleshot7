## Intro

Refund policy is not just a legal note. It is part of whether users can trust the product.

## Why This Matters

An AI tool deals with uploads, queueing, and generation errors. That means failures can happen even when the user did nothing wrong.

## When Refunds Make Sense

### 1. The job never starts

If payment succeeds but the generation pipeline never begins, the user should not be stuck with a dead session.

### 2. The system fails and cannot recover

If the service hits a processing failure and cannot produce a usable result within the supported flow, refunding is the cleaner outcome.

### 3. The platform confirms a backend-side error

Payment, storage, or generation issues caused by the service itself should not be pushed onto the user.

### 4. The policy already promises an automatic refund

Once the rule is published, the service should apply it consistently.

## When Refunds Usually Do Not Apply

- The user uploads a weak or unusable selfie
- The result is stylistically different but technically delivered
- The user changes their mind after a successful run

## Quick Checklist

- Distinguish user error from system error
- Keep the refund rule visible
- Use idempotent refund logic
- Record failure reasons clearly
